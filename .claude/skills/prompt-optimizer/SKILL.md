# Claude Code 프롬프트 최적화 스킬

## 목적
사용자의 자연스러운 명령을 Claude Code의 베스트 프랙티스에 맞게 최적화하여 더 효율적인 결과를 얻도록 돕는다.

## 작동 방식
1. 사용자가 자연스러운 언어로 원하는 작업 설명
2. Claude Code 공식 가이드의 핵심 원칙 적용:
   - **구체성**: 모호한 지시 → 명확한 지시
   - **계획 우선**: 탐색 → 계획 → 구현 → 검증
   - **컨텍스트 관리**: 파일 참조, 관련 코드 명시
   - **검증 단계**: 테스트, 리뷰, 롤백 계획 포함
   - **병렬 처리**: 가능한 경우 여러 에이전트 활용

3. 최적화된 프롬프트 출력
4. 사용자 승인 후 실행 또는 수정

## 최적화 규칙

### 1. 명확성 규칙
**Before**: "이 코드 고쳐줘"
**After**: 
```
@src/components/Button.tsx의 TypeScript 오류를 수정해:
1. 먼저 오류 메시지와 관련된 타입 정의 확인
2. 누락된 props 인터페이스 추가
3. ESLint 오류가 없는지 검증
4. 기존 테스트가 통과하는지 확인
```

### 2. 계획 모드 규칙
**Before**: "새로운 로그인 페이지 만들어"
**After**:
```
로그인 페이지 구현 계획을 먼저 세워줘:

현재 상황:
- @src/pages/ 폼 구조 확인
- @src/components/에서 재사용 가능한 입력 컴포넌트 파악
- @src/auth/ 인증 로직 확인

계획에 포함할 내용:
1. 파일 구조 및 라우트 설계
2. 필요한 컴포넌트 목록 (이메일 입력, 비밀번호 입력, 제출 버튼)
3. 유효성 검사 규칙
4. 에러 핸들링 방식
5. 테스트 전략

계획 검토 후 단계별로 구현하자.
```

### 3. 컨텍스트 제공 규칙
**Before**: "버그 수정해"
**After**:
```
@src/utils/api.ts의 fetch 함수에서 발생하는 버그 수정:

버그 증상:
- 401 에러 시 무한 루프 발생
- 토큰 갱신이 중복으로 실행됨

관련 파일:
- @src/utils/api.ts: API 호출 로직
- @src/hooks/useAuth.ts: 인증 상태 관리
- @src/services/token.ts: 토큰 갱신 로직

수정 후 반드시 @tests/api.test.ts 실행해서 검증해.
```

### 4. 검증 단계 포함 규칙
**Before**: "리팩토링 해줘"
**After**:
```
@src/services/user.ts 리팩토링:

1. 먼저 현재 코드 구조 분석하고 개선점 문서화
2. 점진적 리팩토링 진행:
   - 중복 함수 추출
   - 에러 핸들링 통일
   - 타입 안정성 개선
3. 각 단계마다 실행:
   - TypeScript 컴파일 체크
   - ESLint 검사
   - 단위 테스트 실행
4. 최종 검증:
   - @tests/user.test.ts 전체 통과 확인
   - API 통합 테스트 수동 검증
   - 변경사항 git diff로 리뷰

문제 발생 시 즉시 롤백하고 알려줘.
```

### 5. 병렬 처리 규칙
**Before**: "여러 파일 리뷰해"
**After**:
```
병렬로 코드 리뷰 진행:

리뷰 대상:
- @src/components/Header.tsx: 네비게이션 구조
- @src/components/Footer.tsx: 링크 및 접근성
- @src/components/Layout.tsx: 레이아웃 패턴

각 파일에 대해 독립적으로 검토:
1. TypeScript 타입 안정성
2. 성능 최적화 포인트
3. 접근성 (a11y) 준수 여부
4. 테스트 커버리지

개별 리뷰 결과를 종합해서 우선순위 제안해줘.
```

### 6. 파일 참조 규칙
- 파일 경로는 `@` 접두사로 명시: `@src/components/Button.tsx`
- 디렉토리 참조 시 명확히: `@src/utils/`에 있는 모든 헬퍼 함수
- 관련 파일 그룹화: "@src/types/와 @src/interfaces/의 타입 정의 통일"

### 7. 명령어 구조
```
[작업 유형] + [대상] + [제약사항] + [검증 단계]

작업 유형: 구현 / 수정 / 리팩토링 / 분석 / 검토
대상: @파일경로 또는 컴포넌트/함수명
제약사항: 기존 코드와의 호환성, 성능 요구사항 등
검증 단계: 테스트 실행, 린트 체크 등
```

## 변환 예시

### 예시 1: 단순 버그 수정
**User**: "로그인이 안 돼"
**Optimized**:
```
로그인 문제 디버깅 및 수정:

1. 재현 확인:
   - @src/pages/Login.tsx의 로그인 폼 동작 확인
   - @src/services/auth.ts의 login 함수 검토
   - 브라우저 개발자 도구 Network 탭에서 API 응답 확인

2. 문제 원인 파악 후 수정:
   - 에러 메시지 로깅 추가
   - 상태 관리 로직 점검

3. 검증:
   - 성공/실패 시나리오 각각 테스트
   - @tests/auth.test.ts 실행
   - 타입 체크 통과 확인
```

**Optimized**:
```
다크모드 기능 구현:

사전 확인:

   - CSS 변수 또는 Tailwind dark: 클래스 적용

   - @src/components/ThemeToggle.tsx 생성
   - Header에 토글 버튼 추가

4. 검증:
   - 모든 컴포넌트에서 테마 전환 테스트
```

### 예시 3: 코드 리뷰
**User**: "이 PR 검토해"
**Optimized**:
```
PR 코드 리뷰:

1. 변경사항 파악:
   - git diff로 전체 변경 범위 확인
   - 관련 Issue 또는 요구사항 문서 확인

2. 파일별 세부 리뷰:
   - @src/components/: 컴포넌트 구조 및 props 인터페이스
   - @src/hooks/: 커스텀 훅의 의존성 및 메모이제이션
   - @src/utils/: 순수 함수 및 에러 핸들링
   - @tests/: 테스트 커버리지 및 케이스 적절성

3. 체크리스트:
   - TypeScript 에러 없음
   - 성능 저하 없음 (불필요한 리렌더링 체크)
   - 보안 취약점 없음
   - 기존 테스트 깨지지 않음

4. 리뷰 결과:
   - 필수 수정사항 (blocking)
   - 권장 개선사항 (non-blocking)
   - 칭찬할 점
```

**User**: "이거 정리해"
**Optimized**:
```
코드 정리 및 리팩토링:

대상 범위:

단계별 진행:

   - 유틸리티 함수 추출 위치

```

## 사용법

사용자가 자연스러운 명령을 내리면:

1. 사용자의 의도 파악 (작업 유형 분류)
2. 위 규칙에 따라 최적화된 프롬프트 생성
3. 사용자에게 변환된 프롬프트 제시:
   - "다음과 같이 요청하면 더 효과적입니다:"
   - 최적화된 프롬프트 코드 블록으로 표시
   - 각 개선점에 대한 간략한 설명
4. 사용자 선택:
   - "이대로 실행" → 변환된 프롬프트로 작업 시작
   - "수정 후 실행" → 사용자가 수정한 버전으로 실행
   - "원래대로" → 사용자의 원래 명령 그대로 실행

## 주의사항
- 과도한 최적화 지양: 간단한 작업은 간단하게
- 사용자 의도 존중: 최적화가 사용자의 실제 필요와 다를 수 있음
- 유연성 유지: 가이드라인이지 절대 규칙은 아님
